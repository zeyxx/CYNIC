# D2 — LLM Inference & Performance Domain — Axiom Evaluation Criteria

Evaluate the TECHNICAL RIGOR of claims about LLM inference, hardware, models, and serving. Not engagement, not popularity. A low-follower engineer posting measured tok/s numbers with hardware specs is HIGH-FIDELITY. A viral thread saying "this model is amazing" with no measurements is noise.

Judge WHETHER THIS CONTENT WOULD HELP AN ENGINEER MAKE BETTER INFERENCE DECISIONS. Benchmarks, configurations, failure reports, and measured tradeoffs are signal. Hype, marketing, and unverified claims are noise.

## ASYMMETRIC CONFIDENCE

Measured data > theoretical claims. Failure reports > success reports (negative results are rarer and more informative). Reproducible benchmarks with hardware context > isolated screenshots.

- Unverified claim ("this model is fast"): max axiom score = 0.25
- Claim with partial numbers ("got 50 tok/s"): max axiom score = 0.40
- Measured benchmark with hardware + config + methodology: max axiom score = 0.618

## FIDELITY
Does this content contain ACCURATE, MEASURABLE claims about inference performance?
- HIGH (0.50-0.618): Reports specific metrics (tok/s, TTFT, p99 latency) on named hardware (RTX 4090, M3 Ultra, H100) with named model (Qwen3.6-27B, Llama 3.1 70B) and quantization (Q4_K_M, IQ3_XXS, FP8). Configuration documented (context length, batch size, parallel slots). Example: "Qwen3.6-27B IQ3_XXS on RTX 4060 Ti: 19.7 tok/s gen, 523 tok/s prompt, ctx=65536, parallel=1, q4_0 KV."
- MEDIUM (0.25-0.45): Names model and hardware but incomplete metrics. General benchmark comparison without exact configuration. Reports improvement without baseline. Example: "Llama 3.1 runs great on my 3090 with vLLM, much faster than before."
- LOW (0.05-0.20): Vague claims without specific models, hardware, or numbers. Marketing language ("blazing fast", "revolutionary performance"). Screenshot of a benchmark without context. Example: "This new model is the fastest ever!"

## PHI
Is this content structurally sound? Does it present a complete picture or cherry-picked data?
- HIGH (0.50-0.618): Compares multiple configurations or engines on the same hardware. Reports both prefill AND decode speeds. Acknowledges tradeoffs (speed vs quality, VRAM vs context). Methodology documented. Example: "Tested vLLM vs SGLang on H100: vLLM wins throughput at batch=32, SGLang wins at p99 latency. Both at FP8, same prompt distribution."
- MEDIUM (0.25-0.45): Single configuration tested but with clear metrics. Acknowledges limitations. Partial comparison (one engine, one model). Example: "llama.cpp b9180 MTP gave me 1.85x on my 3090 — haven't tested other engines yet."
- LOW (0.05-0.20): Cherry-picked single number without context. Best-case-only reporting. Ignores obvious tradeoffs (reports speed, ignores quality loss from heavy quantization). Example: "10000 tok/s!!!" (batch=256, short prompts, no p99).

## VERIFY
Can the performance claims be INDEPENDENTLY REPRODUCED?
- HIGH (0.50-0.618): Provides exact model file (HF repo, GGUF filename), exact engine version (commit hash, release tag), exact hardware, exact flags/config. Anyone with the same setup can reproduce. Example: "bartowski/Qwen3.6-27B-IQ3_XXS.gguf on llama.cpp b9180 with --spec-type draft-mtp --spec-draft-n-max 2 --flash-attn on RTX 3090."
- MEDIUM (0.25-0.45): Provides most components but missing one (no engine version, no exact quant). Reproducible with some guessing. Example: "Qwen 27B on llama.cpp, Q4 quantization, got 25 tok/s on my 4090."
- LOW (0.05-0.20): No reproducibility path. Screenshot without config. Anecdotal ("feels faster"). Impossible to verify. Example: "My AI runs much faster now after the update."

## CULTURE
Does this follow established benchmarking and reporting norms?
- HIGH (0.50-0.618): Follows community standards: reports hardware specs, engine versions, model details. Discloses affiliations (works at NVIDIA, maintains vLLM). Cites prior work. Engages with corrections. Example: HuggingFace community member posting on the Open LLM Leaderboard with full methodology.
- MEDIUM (0.25-0.45): Generally honest reporting but informal. Twitter-length constraints limit detail. Author has history of technical posts. Example: Regular ML engineer sharing weekend benchmarks.
- LOW (0.05-0.20): Marketing disguised as benchmarks. Company posting best-case numbers for their product. Astroturfing. No disclosure of conflicts. Example: GPU vendor posting "our new chip is 10x faster" with cherry-picked workload.

## BURN
Is this content efficient? Maximum signal per word?
- HIGH (0.50-0.618): Dense with numbers and specifics. Table format for comparisons. No filler. A single tweet that captures hardware+model+config+result. Example: "3x 1080 Ti + llama.cpp MTP. Qwen 3.6 27B: 29.9 t/s @ 196K ctx."
- MEDIUM (0.25-0.45): Good signal but padded with explanation or context that could be shorter. Thread format where 1-2 tweets would suffice. Example: 5-tweet thread where tweets 1 and 5 are the signal, 2-4 are filler.
- LOW (0.05-0.20): High word count, low information density. Repeats the same claim multiple ways. Excessive emojis, caps, engagement bait wrapping thin content. Example: "HUGE BREAKTHROUGH! You won't BELIEVE how fast this model runs! 🚀🔥 (15 tweets of hype, one number in tweet 12)."

## SOVEREIGNTY
Does this information preserve engineering autonomy?
- HIGH (0.50-0.618): Reports on open-source models, open engines, reproducible setups. Helps engineers run inference on their own hardware. Compares self-hosted vs cloud options. Example: "Here's how to run 27B on consumer hardware with full context — no API dependency."
- MEDIUM (0.25-0.45): Mixed — reports on both proprietary and open solutions. Useful information but requires vendor lock-in to act on. Example: "TensorRT-LLM gives best H100 perf but requires NVIDIA stack."
- LOW (0.05-0.20): Promotes vendor lock-in without alternatives. Benchmarks only proprietary solutions. Requires paid API to replicate. Example: "Just use GPT-5 API, local inference is dead."
